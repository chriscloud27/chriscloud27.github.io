Title: ADR-0007: GitHub Actions Generates Root Locale Redirect at Deploy Time

Status: accepted

Date: 2026-04-22

## Context

Next.js middleware handles locale detection at runtime, but static export (ADR-0006) eliminates middleware. Without a root `index.html` that redirects to a locale prefix, visiting `mach2.cloud/` returns a 404 on GitHub Pages.

Two approaches for generating this redirect were considered:

**Option A — Commit a static `index.html` to the repo root or `public/` folder:** Simple — the file is versioned alongside the code. Risk: the redirect logic (supported locales, default) can fall out of sync with `i18n/routing.ts` if one is changed without the other.

**Option B — Generate `out/index.html` during the GitHub Actions build step (after `next build`):** The CI workflow writes the file with `cat > ./out/index.html` immediately after the build completes. The file is never committed to the repository. Supported locales and default locale are always written fresh from the CI script, which references the same constants as the app.

Option B was chosen to keep the redirect generation close to the build step, prevent a stale committed file from diverging from the app's actual locale configuration, and avoid adding a non-Next.js HTML file to the source tree.

The generated file reads `navigator.language`, matches it against the supported locales array (`['en', 'de', 'es']`), and calls `window.location.replace()`. A `<noscript>` meta-refresh fallback redirects to `/en/`.

## Decision

The root `out/index.html` locale redirect is generated by a `cat > ./out/index.html` heredoc in the GitHub Actions deploy workflow, written after `npm run export`. It is not a source file in the repository. The supported locales array in the CI script must be kept in sync with `i18n/routing.ts`.

## Consequences

- When adding or removing a locale (see ADR-0009), the supported locales array in `.github/workflows/deploy.yml` must be updated at the same time as `i18n/routing.ts`. The two files have no compile-time link.
- The redirect uses `window.location.replace()` (not `assign`) so the blank redirect page is not added to browser history.
- The `<noscript>` fallback always redirects to `/en/` — it does not detect language. This is acceptable since `<noscript>` is a last resort for JavaScript-disabled environments.
- `.nojekyll` is also created in the same CI step — do not move the `cat` heredoc without also preserving the `touch ./out/.nojekyll` line.
