Title: ADR-0002: TypeScript as the Primary Language for All Application Code

Status: accepted

Date: 2026-04-22

## Context

The project could be written in plain JavaScript or TypeScript. Two approaches were considered:

**Option A — JavaScript:** No compile step for type checking. Faster initial setup. Lower barrier for one-off scripts. However, no type safety for component props, API response shapes, SEO config structures, or i18n message keys. Errors in data shapes (e.g. missing `locale` in params) surface at runtime, not at build time.

**Option B — TypeScript:** Compile-time type checking. IDE autocomplete for component props, `SITE_CONFIG` fields, `next-intl` message keys, and `Metadata` objects. Errors caught before deploy. Next.js 15 ships with TypeScript-first APIs — the `Metadata` type, `generateStaticParams` return type, and `params: Promise<{ locale: string }>` pattern all rely on TypeScript inference for correctness.

TypeScript was chosen because the cost of type errors in this codebase is high: a wrong locale type breaks all pages; a wrong `sameAs` URL breaks structured data for all locales. The compile-time feedback loop is worth the setup overhead for a project that builds on top of type-heavy APIs (Next.js, next-intl, TanStack Query).

**Exception:** `lib/compassEngine.js` and `lib/compassReports.js` are intentionally kept as plain JavaScript. These modules must run outside the Next.js build system (in n8n, CLI scripts) where TypeScript compilation is not available. See ADR-0011.

## Decision

All application code under `app/`, `components/`, `lib/` (except Compass modules), `i18n/`, and `scripts/` is written in TypeScript. The Compass engine and report builder remain plain JavaScript for framework-agnostic portability.

## Consequences

- All new source files must use `.ts` or `.tsx` extensions unless they are intentionally framework-agnostic (see ADR-0011 exception).
- `tsconfig.json` uses strict mode — do not disable `strictNullChecks` or `noImplicitAny` to work around a typing issue. Fix the type instead.
- `next-env.d.ts` is auto-generated by Next.js and must not be edited manually.
- Type errors are build-blocking — CI will fail if TypeScript compilation fails. This is intentional.
- `scripts/*.mjs` files use `.mjs` (ES module JavaScript) because they run directly in Node.js without a build step. These are tooling scripts, not application code.
